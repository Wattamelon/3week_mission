# Mini NPU Simulator

## 1. 프로젝트 소개

이 프로젝트는 입력 패턴과 필터 간의 유사도를 **MAC(Multiply-Accumulate)** 연산으로 계산하여,  
입력된 패턴이 **Cross(십자가)** 와 **X** 중 어떤 패턴과 더 유사한지 판별하는 Python 콘솔 프로그램이다.

사람은 모양을 직관적으로 인식할 수 있지만, 컴퓨터는 숫자 배열만 처리할 수 있기 때문에  
패턴과 필터를 2차원 배열로 표현하고, 위치별 곱셈 후 합산하는 방식으로 유사도를 계산한다.

본 프로젝트에서는 다음 두 가지 모드를 구현하였다.

- **Mode 1**: 사용자 입력 기반 3x3 패턴 판별
- **Mode 2**: `data.json` 기반 자동 판별 및 성능 분석

4. 구현 요약
4-1. MAC 연산 구현

MAC(Multiply-Accumulate) 연산은 입력 패턴과 필터의 같은 위치 값끼리 곱한 뒤,
그 결과를 모두 더하여 최종 점수를 계산하는 방식이다.

예를 들어 3x3 패턴과 필터가 있을 때:
0 1 0
1 1 1
0 1 0

각 위치를 곱한 뒤 모두 더하면 유사도 점수가 된다.
점수가 높을수록 해당 필터와 더 유사하다고 판단한다.

4-2. 입력 검증 방식

Mode 1에서는 사용자 입력을 직접 받기 때문에 다음 검증을 수행하였다.

각 줄에 정확히 n개의 숫자가 입력되었는지 확인
입력값이 0 또는 1인지 확인
잘못된 입력이 들어오면 오류 메시지를 출력하고 재입력 유도

이를 통해 프로그램이 비정상 종료되지 않도록 구성하였다.

4-3. 라벨 정규화 방식

Mode 2의 data.json에서는 정답 라벨이 다음과 같이 표현된다.

"+" → Cross
"x" → X

프로그램 내부에서는 비교를 일관되게 수행하기 위해
모든 라벨을 표준 라벨인 Cross, X로 정규화하여 처리하였다.

4-4. 동점 처리 정책

두 필터의 점수가 매우 비슷하거나 같은 경우를 처리하기 위해
허용오차(epsilon) 기반 비교를 사용하였다.

abs(score_a - score_b) < 1e-9

위 조건을 만족하면 UNDECIDED로 판정한다.

이 방식은 부동소수점 연산 오차로 인해
실제로는 같은 값인데도 다르게 판정되는 문제를 줄이기 위해 사용하였다.

5. Mode 1 설명

Mode 1에서는 사용자가 직접 다음 값을 입력한다.

3x3 필터 A
3x3 필터 B
비교할 3x3 패턴

입력 후 프로그램은 다음을 수행한다.

패턴과 필터 A의 MAC 점수 계산
패턴과 필터 B의 MAC 점수 계산
두 점수 비교 후 A / B / UNDECIDED 판정
평균 연산 시간(ms) 출력

6. Mode 2 설명

Mode 2에서는 data.json 파일에 저장된 여러 패턴을 자동으로 판별한다.

처리 흐름
data.json 파일 로드
filters와 patterns 읽기
patterns의 키에서 크기 정보 추출
예: size_13_2 → 13
해당 크기의 필터 선택
size_13 필터 사용
Cross / X 필터와 각각 MAC 점수 계산
판정 결과와 expected 비교
PASS / FAIL 출력
전체 요약 출력
7. 결과 리포트
7-1. 실행 결과 요약

예시 실행 결과:
총 테스트: 9개
통과: 6개
실패: 3개

7-2. FAIL 케이스 원인 분석

본 프로젝트에서는 일부러 data.json에 FAIL 케이스도 포함시켜
예외 상황을 처리할 수 있도록 구현하였다.

실패 원인은 크게 다음과 같이 분류할 수 있다.

(1) 패턴 크기 불일치

예를 들어 키 이름은 size_5_3인데 실제 입력 패턴은 3x3인 경우,
프로그램은 이를 5x5 패턴으로 기대하지만 실제 크기가 다르므로 FAIL 처리하였다.

(2) 행 길이 불일치

일부 패턴에서 특정 행의 길이가 기대한 열 개수와 다를 경우,
정상적인 행렬로 간주할 수 없기 때문에 FAIL 처리하였다.

(3) 동점(UNDECIDED) 처리

Cross와 X 필터 점수가 거의 같을 경우
epsilon 기준에 따라 UNDECIDED로 판정되며,
정답 라벨과 일치하지 않으면 FAIL로 처리된다.

(4) 라벨 비교 문제

정답 라벨이 "+", "x" 형태로 저장되어 있기 때문에
이를 Cross, X로 정규화하지 않으면 비교가 정확히 되지 않는다.
따라서 라벨 정규화가 없었다면 정상 케이스도 FAIL이 발생할 수 있다.

7-3. 왜 프로그램이 비정상 종료되지 않도록 했는가

Mode 2에서는 잘못된 데이터가 존재할 수 있으므로,
한 케이스에서 오류가 발생하더라도 프로그램 전체가 종료되지 않고
해당 케이스만 FAIL 처리 후 다음 케이스로 넘어가도록 구현하였다.

이 방식은 실제 데이터 처리 상황에서도 매우 중요하며,
전체 자동 테스트를 안정적으로 수행할 수 있게 해준다.

8. 성능 분석
8-1. 측정 방식

성능 분석은 I/O(입력/출력, 파일 읽기)를 제외하고
MAC 연산 함수 호출 구간만 측정하도록 구성하였다.

각 크기별로 최소 10회 반복 측정 후 평균값을 구하였다.

8-2. 예시 성능 결과
크기       평균 시간(ms)    연산 횟수
-----------------------------------
5x5        0.010000         25
13x13      0.030000         169
25x25      0.090000         625

8-3. 시간 복잡도 분석

MAC 연산은 패턴과 필터의 각 위치를 한 번씩 모두 순회하며 계산한다.

즉, N x N 크기의 패턴에 대해
총 연산 횟수는 다음과 같다.
N² , O(N²)

예를 들어:

3x3 → 9회 연산
5x5 → 25회 연산
13x13 → 169회 연산
25x25 → 625회 연산

즉, 패턴 크기가 커질수록 연산 횟수가 제곱 형태로 증가하므로
실제 측정 시간도 함께 증가하는 경향을 보인다.

이를 통해 단순한 패턴 판별이라도
입력 크기가 커질수록 계산 비용이 빠르게 증가함을 확인할 수 있었다.

10. 느낀 점

이번 프로젝트를 통해 단순한 패턴 판별 문제를 통해서도
AI 연산의 핵심인 MAC 연산 구조를 직접 구현해볼 수 있었다.

특히 단순히 계산만 구현하는 것이 아니라,

입력 검증

자동 테스트
성능 분석

까지 포함해야 한다는 점에서
단순 알고리즘 구현보다 한 단계 더 실제적인 프로그램 구조를 경험할 수 있었다.

또한 패턴 크기가 증가할수록 연산량이 빠르게 증가한다는 점을 통해
왜 AI 전용 하드웨어(NPU)가 필요한지도 간접적으로 이해할 수 있었다.
